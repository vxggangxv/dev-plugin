# 개발 에이전트 공통 워크플로우

모든 개발 에이전트는 코딩 전에 반드시 이 워크플로우를 따른다.

---

## Phase 1: READ (문서 정독)

제공된 모든 문서를 읽고 핵심 정보를 파악한다.

1. **spec.md** 읽기 - 요구사항, Acceptance Criteria
2. **plan.md** 읽기 - 구현 전략, 마일스톤, 작업 분해
3. **작업 대상 파일** 확인 - 지정된 파일 경로가 실제로 존재하는지 확인
4. **기존 코드** 읽기 - 대상 파일의 현재 코드를 반드시 읽고 패턴 파악

## Phase 2: EXTRACT (요구사항 추출)

문서에서 **모든 구체적 요구사항**을 번호 매긴 체크리스트로 추출한다.
추출 시 다음 항목을 하나도 빠뜨리지 않는다:

- [ ] **파일**: 작업 대상 파일 경로 (모든 파일 나열)
- [ ] **파라미터**: API 파라미터, props, 함수 인자 (이름, 타입, 필수 여부)
- [ ] **엔드포인트**: API 경로, HTTP 메서드
- [ ] **UI 요소**: 버튼, 입력 필드, 텍스트 (정확한 문구 포함)
- [ ] **로직**: 조건 분기, 유효성 검증, 에러 처리
- [ ] **데이터**: 모델 필드, 스키마 변경, 타입 정의
- [ ] **연동**: 다른 모듈/API와의 연결 지점

추출 결과를 다음 형식으로 작성:

```
## 요구사항 체크리스트

- [ ] #1: [구체적 요구사항]
- [ ] #2: [구체적 요구사항]
- [ ] #3: [구체적 요구사항]
...
```

## Phase 3: CODE (구현)

체크리스트를 보면서 **항목별로** 구현한다.

- 각 항목 구현 완료 시 체크 표시
- 한 파일씩 순서대로 작업 (여러 파일 동시 수정 금지)
- 대상 파일이 지정되었으면 **반드시 해당 파일에서** 작업
- 기존 코드 패턴을 따르기 (import 스타일, 네이밍, 구조)

## Phase 4: VERIFY (자기 검증 + 자가 개선 루프)

구현 완료 후 다음을 반드시 수행한다.

### 4-1. 체크리스트 대조
Phase 2에서 추출한 체크리스트를 하나씩 다시 확인한다:
- 모든 항목이 체크되었는가?
- 빠뜨린 파라미터/필드/UI 요소가 없는가?

### 4-2. 원본 문서 재확인
spec.md/plan.md를 **다시 한번 읽고** 구현 결과와 대조한다:
- 문서에 명시된 내용 중 구현에 반영되지 않은 것이 있는가?
- 문서에 없는 내용을 임의로 추가하지 않았는가?

### 4-3. 대상 파일 확인
- 지정된 모든 파일에 작업이 완료되었는가?
- 수정하면 안 되는 파일을 건드리지 않았는가?

### 4-4. 자가 개선 루프 (Self-Eval Loop)

**테스트/검증이 모두 통과할 때까지 아래 루프를 반복한다.**

```
┌───────────────────────────────────────────┐
│  1. 실행: 테스트 / 타입체크 / 린트 실행     │
│              ↓                             │
│  2. 판정                                   │
│     - 전부 통과 → 루프 종료 ✅              │
│     - 실패 → 3단계로                        │
│              ↓                             │
│  3. 실패 원인 분류 & 수정                    │
│     a) 테스트 코드 문제 → 테스트 수정        │
│     b) 소스 코드 버그 → 소스 수정            │
│     c) 환경/설정 문제 → 설정 수정            │
│     d) flaky → locator/대기/격리 개선       │
│              ↓                             │
│  4. 수정 적용 후 1번으로 돌아감               │
└───────────────────────────────────────────┘
```

**루프 규칙:**
- **최대 반복 횟수: 5회** — 5회 내 해결 불가 시 실패 원인과 시도 내역을 보고하고 중단
- **매 반복마다 수정 내역 기록** — 어떤 문제를 어떻게 수정했는지 추적
- **같은 원인으로 2회 연속 실패 시** — 접근 방식을 변경
- **소스 코드 수정 가능** — 테스트뿐 아니라 소스 코드의 실제 버그도 수정

### 4-5. 누락 발견 시
누락된 항목이 있으면 **즉시 수정**하고, 수정 내용을 보고한다.

---

## 완료 보고 형식

```markdown
## 작업 완료 보고

### 요구사항 체크리스트
- [x] #1: [완료된 항목]
- [x] #2: [완료된 항목]
- [x] #3: [완료된 항목]

### 수정된 파일
- [파일 경로]: [변경 내용 요약]

### 검증 결과
- 체크리스트 대조: 전체 N개 중 N개 완료
- 원본 문서 재확인: 누락 없음 / [누락 항목 및 수정 내역]

### 주의사항
- [추가 확인이 필요한 사항]
```
