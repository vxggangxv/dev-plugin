# Backend Development Workflow

**Tidy First** 원칙 기반 백엔드 개발 워크플로우.

---

## 1. 핵심 원칙

### 1.1 Tidy First 접근법

모든 변경사항을 두 가지 유형으로 분리:

| 유형 | 설명 | 예시 |
|------|------|------|
| **구조적 변경** | 동작 변경 없이 코드 재배치 | 이름 변경, 메서드 추출, 코드 이동 |
| **동작 변경** | 실제 기능 추가/수정 | 새 기능 구현, 버그 수정 |

**핵심 규칙**
- 구조적 변경과 동작 변경을 같은 커밋에 혼합하지 말 것
- 둘 다 필요할 때는 항상 **구조적 변경을 먼저** 수행
- 구조적 변경 전후로 테스트를 실행하여 동작 변경이 없음을 검증
- 한 번에 하나의 리팩토링만 수행, 각 단계 후 테스트 실행

### 1.2 코드 품질 표준

| 원칙 | 설명 |
|------|------|
| **중복 제거** | 철저하게 중복 제거 |
| **의도 표현** | 명명과 구조를 통해 의도 명확히 표현 |
| **의존성 명시** | 의존성을 명시적으로 만들기 |
| **작은 메서드** | 단일 책임에 집중하는 작은 메서드 유지 |
| **상태 최소화** | 상태와 부작용 최소화 |
| **단순함** | 가능한 가장 간단한 해결책 사용 |

### 1.3 구현 전 사고

- 구현 전에 가정(assumptions)을 명시적으로 밝히고, 불확실하면 질문
- 여러 해석이 가능하면 선택지를 제시 — 임의로 선택하지 않기
- 더 단순한 접근법이 있으면 말하기. 필요하면 반론(push back) 제시
- 불명확한 부분이 있으면 멈추고, 무엇이 불명확한지 명시하고 질문

### 1.4 단순성 우선

- 요청받지 않은 기능을 추가하지 않기
- 한 번만 사용되는 코드에 추상화를 만들지 않기
- 불가능한 시나리오에 대한 에러 처리를 추가하지 않기
- 200줄로 작성한 코드가 50줄로 가능하면 다시 작성

### 1.5 최소 변경 원칙

- 인접 코드, 주석, 포맷팅을 "개선"하지 않기
- 고장나지 않은 것을 리팩토링하지 않기
- 기존 스타일을 따르기 (본인 스타일과 다르더라도)
- 본인의 변경으로 사용되지 않게 된 import/변수/함수는 제거
- 기존에 있던 데드 코드는 요청 없이 제거하지 않기

## 2. 워크플로우: 플랜 → 작업 → 테스팅

### 2.1 Context Isolation Protocol

- 한 번에 **하나의 기능(Feature)**과 **하나의 단계(Phase)**에만 집중
- 대화 시작 시 기능이 지정되지 않았다면 **"어떤 기능을 작업하시겠습니까?"** 먼저 질문
- 오직 지정된 `specs/[feature-name]/` 폴더 내의 파일만 Source of Truth로 취급

### 2.2 Dynamic Context Loading

작업 시작 전 반드시 다음 순서로 해당 기능 폴더의 파일을 로드:

| 순서 | 파일 | 목적 |
|------|------|------|
| 1 | `specs/[feature]/spec.md` | Goal - 변경 불가한 요구사항 |
| 2 | `specs/[feature]/plan.md` | Map - 현재 단계 확인 |
| 3 | `specs/[feature]/context.md` | State - 이전 세션의 마지막 상태 |

### 2.3 Phase-Based Implementation

#### Phase 1: 플랜 (No Code)
- 코드 작성 전 `plan.md` 생성 또는 업데이트
- 구현을 3~5개의 작은 단계(Phase)로 분리
- 승인 대기: 계획 수립 후 사용자 승인 없이 코드 작성 금지

#### Phase 2: 작업 (Task by Task)
- 체크리스트를 하나씩 수행
- 한 번에 하나의 파일/함수만 수정
- 구조적 변경과 동작 변경 분리 커밋

#### Phase 3: 테스팅
- 구현된 기능에 대한 **테스트 코드 작성**
- **Green이 되도록** 테스트와 코드를 조정
- 모든 테스트 통과 확인 후 Phase 완료

#### Phase 4: Completion & Archiving
1. `context.md`에 완료된 내용 요약 기록
2. `plan.md`의 해당 단계를 완료(Done) 처리
3. Self-Correction: 다음 단계 진행 전 `spec.md` 요구사항 위반 여부 검토

### 2.4 Documentation Maintenance

모든 작업 진행 상황은 `specs/[feature]/` 폴더 아래의 문서에 실시간 반영.

#### context.md 업데이트 규칙
1. 코드 변경 시마다 `context.md`를 최신 상태로 유지
2. 포함 항목: 현재 상태, 완료된 작업, 다음 작업, 중요 결정/발견, 변경된 파일
3. 세션 종료 전 반드시 최종 상태 요약 기록
4. 새 세션이 이 파일만 읽고도 중단된 지점부터 바로 재개 가능해야 함

---

## 3. 새 기능 워크플로우

```
1. 플랜: plan.md 작성 → 사용자 승인
       ↓
2. 작업: 최소한의 코드 구현 (한 번에 하나씩)
       ↓
3. 필요한 구조적 변경 수행 (Tidy First) → 별도 커밋
       ↓
4. 테스팅: 테스트 코드 작성 → Green이 되도록 조정
       ↓
5. 커밋 (구조적 변경과 동작 변경 별도)
       ↓
6. 다음 기능 증분에 대해 반복
```

---

## 4. 커밋 규율

- 모든 테스트 통과 + 린터 경고 해결 + 단일 논리적 작업 단위일 때만 커밋
- 구조적 변경인지 동작 변경인지 명확히 명시
- 작고 빈번한 커밋 선호
