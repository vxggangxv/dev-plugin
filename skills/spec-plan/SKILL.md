# Spec Plan 스킬

사용자의 작업 요청을 분석하고, 불명확한 부분을 질문으로 확인한 후 구현 계획을 수립한다.

**워크플로우 참조**: `~/.claude/skills/spec/references/workflow.md`

---

## 동작 방식

### 1. 요청 분석

사용자 입력을 분석하여 다음을 파악한다:
- 무엇을 만들거나 수정하려는지
- 어떤 기술/파일/모듈이 관련되는지
- 명확한 부분과 불명확한 부분 구분

### 2. 코드 기반 검증

요청과 관련된 코드를 탐색하여 분석 내용을 검증한다:
- 언급된 파일/모듈이 실제로 존재하는지
- 현재 코드 구조와 요청의 정합성
- 수정 대상의 실제 구현 상태

검증 결과를 사용자에게 먼저 공유한다:

```
## 코드 검증 결과

**확인된 현재 상태:**
- [파일/모듈 존재 여부, 현재 구조 등]

**요청과의 차이점:**
- [예상과 다른 부분이 있다면]
```

이 검증 결과를 바탕으로 이후 질문과 계획을 수립한다.

### 3. 질문 (불명확한 부분이 있을 때)

불명확한 부분이 있으면 **반드시 먼저 질문**한다. 질문 없이 추측하여 진행하지 않는다.

질문 형식:
```
요청을 분석했습니다.

**이해한 내용:**
- [명확하게 파악된 부분]

**확인이 필요한 부분:**
1. [질문 1]
2. [질문 2]
3. [질문 3]

답변해주시면 구체적인 계획을 세우겠습니다.
```

질문 기준:
- 대상 범위가 불명확 (어떤 페이지? 어떤 API?)
- 기술적 선택이 필요 (라이브러리, 패턴, 구조)
- 비즈니스 로직이 불명확 (조건, 예외 케이스)
- 기존 코드와의 관계가 불명확

불명확한 부분은 **갯수 제한 없이 전부** 물어본다.

### 4. 분기

- **질문이 있을 경우** → 질문하고 답변 대기
- **질문이 없을 경우** → 바로 spec-execute 스킬을 실행

### 5. 답변 수신 후

답변을 반영하여 구현 계획을 제시한다:

```
## 구현 계획

### 목표
[한 줄 요약]

### 작업 목록
1. [작업 1] - [대상 파일/모듈]
2. [작업 2] - [대상 파일/모듈]
3. [작업 3] - [대상 파일/모듈]

### 기술적 접근
- [선택된 방식과 이유]

### 예상 영향
- [수정되는 파일/모듈]
- [주의할 점]
```

계획 승인 후 spec-write 스킬로 spec.md를 생성하고 바로 spec-execute 스킬을 이어서 실행한다.

---

## 핵심 원칙

- **추측하지 않는다** - 모르면 물어본다
- **질문은 간결하게** - 갯수 제한 없이 불명확한 부분 전부 물어본다
- **불명확한 부분이 없으면 질문 없이 바로 실행** - 불필요한 질문 금지
